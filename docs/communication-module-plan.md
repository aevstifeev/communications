# Communication Module Plan

## Goal

Подготовить несколько параллельных прототипов того, как функционал созвонов может жить в продукте, не теряя доступ к текущей реализации.

## Working Model

- `index.html` остается стабильной рабочей точкой входа.
- `concepts/communication/baseline/` хранит текущую версию в ее исходной форме.
- Новые гипотезы по модулю создаются как отдельные папки внутри `concepts/communication/`.
- `concepts/communication/hub.html` используется как страница сравнения вариантов.

## Invariants

- Не ломаем текущий основной сценарий ради экспериментов.
- Общие токены, базовые стили и повторно используемые паттерны остаются в `styles/` и `scripts/`.
- Внутри варианта храним только то, что специфично для этой гипотезы.

## Prototype Flow

1. Сохраняем текущую реализацию в `baseline/`.
2. Создаем конкурентные варианты.
3. Для каждого варианта формулируем гипотезу и критерии проверки.
4. Сравниваем варианты через `hub.html` и документ с описанием.
5. После выбора лучшего решения переносим его в основной продуктовый поток.

## Naming Rule

- Используем смысловые имена, а не `test`, `new`, `final2`.
- Технические папки можно оставить прежними, если это не мешает структуре.
- Смысловые названия вариантов фиксируем в интерфейсе и документации.

## Ready-To-Use Variants

- `baseline` - как есть сейчас;
- `variant-inline-panel` - вариант A: созвоны как часть функционала чатов, каналов и текущей коммуникации;
- `variant-thread-workspace` - вариант B: созвоны как отдельный модуль коммуникации, встроенный в систему наравне с чатами и календарем;
- `variant-context-drawer` - вариант C: резервный, сейчас неактивен.
